# /dev-workflow スキル

## Goal

実装タスクを「探索 → 計画 → 実装 → 検証 → コミット」の流れで進め、検証（テスト・リンタ）を飛ばさずに品質を担保する。新規開発と既存改修で手順を使い分ける。

## Principle

常に自分の作業を検証する。テストとリンタを通さずにタスクを完了扱いにしない。これがコード品質に対する最もレバレッジの高い実践。

## 新規プロダクト開発（new product）

1. **explore**: Plan Mode で既存コードと構造を読み、関連モジュール・依存・既存パターンを把握する。このフェーズではコードを書かない。
2. **plan**: 詳細な実装計画を立てる。テスト戦略（対象・エッジケース・期待挙動）を定義し、適用するリンタ・型チェッカを特定する。着手前にユーザー確認を取る。
3. **implement**: 計画に沿って実装する。各単位の実装と同時か直後にテストコードを書く。大きな一括変更より小さな増分変更を優先する。
4. **verify**: テストを実行して全て通ることを確認。リンタ（eslint, ruff, golangci-lint 等）と型チェッカ（tsc, mypy, pyright 等）を実行し全エラー・警告を解消。失敗したら直して全て通るまで再実行する。
5. **commit**: 変更を要約した説明的なメッセージでコミットする。

## 既存プロダクト改修（existing product）

1. **explore**: 関連コード・git 履歴・既存テストを読み、現挙動を理解。問題の根本原因や変更範囲を特定する。コードは書かない。
2. **plan**: blast radius を最小化し不要な変更を避ける計画を立てる。影響範囲をカバーする既存テストと、必要な新規/修正テストを洗い出す。
3. **test_first**: 変更前に、バグを再現するか新挙動を検証する失敗するテストを書く。実行して、期待どおりの理由で失敗することを確認する。
4. **implement**: 問題修正・改修に必要な最小限の変更のみ行う。無関係なリファクタリングはしない。
5. **verify**: 失敗していたテストが通ることを確認。関連テスト一式を回して回帰がないことを確認。リンタ・型チェッカを実行し全エラーを解消。全て通るまで再実行する。
6. **commit**: 根本原因と修正内容を説明するメッセージでコミットする。

## 検証ルール（共通）

- 実装後は必ずテストを実行する。スキップしない。
- 実装後は必ずリンタを実行し、コミット前に全エラーを解消する。
- 型チェックが構成されているプロジェクトでは実行し全エラーを解消する。
- test/lint/typecheck コマンドが不明なら、まず `package.json` / `Makefile` / `pyproject.toml` 等の設定ファイルを確認する。
- テスト失敗時は根本原因に対処する。エラー抑制やテストスキップで逃げない。
- 既存テストがないプロジェクトでは、プロジェクトの慣習に従ってテストファイルを作る。

## Notes

- リポジトリ常設の lint hook（`lint-changed.sh`）が編集完了時に対象リンタを自動実行し、検証漏れを可視化する。本スキルの verify はそれと併用する。
- 言語別の lint/format コマンドは触れたファイルに応じて path-scoped rules（`.claude/rules/`, `~/.claude/rules/`）が読み込む。
